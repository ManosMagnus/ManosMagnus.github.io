WebSocket Active Scan

Introduction   intro

This project aims to support for Active Scan to WebSocket extension of ZAP. WebSockets is a very recent protocol, which was standardized in 2011. One of the basic features of WebSocket is establishing a connection over TCP and remaining open. In that way it allows the server and the client to create a full-duplex communication channel, thus allowing both of them to truly communicate asynchronously. This it the biggest advantage of WebSocket contrary to traditional Ajax Request/Response. That is the reason the WS is suitable for chats, messages notifications, monitoring etc. Furthermore, recent years WS potocol was used widely over the web applications. With the current version of ZAP we are able to intercept and show WS payloads, set breakpoint on specific types of WS’s payloads and fuzz payloads. Also ZAP WS add-on is considered as a reliable tool for WS communication analysis and debuging. I strongly believe that active scan extension of ZAP will be make it an even more powerful tool.

But which are the vulnerabilities; And what the active scans we can make for; Active scans include tests to web application in Error1, Blind2 and Login by pass3 SQL Injections vulnerabilities. To accomplish our scans we can test web application using Time Based techniques4. The scans will be able to test either specific or already established connections. At this moment, we are not able to automatically establish connection with AjaxSpider. So, my long term goal about that is to establish and scan all websocket connections under a specific endpoint.

Another significant clue about websocket vulnerabilities is that the WebSocket protocol doesn’t handle authentication. Practically this means that, a WebSocket opened from a page behind authentication doesn’t “automatically” receive any sort of authentication; you need to take steps to also secure the WebSocket connection. That makes possible to establish connection without authentication making use of http and origin headers field of the client. WebSocket is a nightmare because it does not follow Same-origin Policy browser restrictions. That vulnerability was known as WebSocket Hijacking.

Some others possible scans at WebSockets are the classic bruteforce attack5, Command execution6, Local7 and Remote8 file inclusion. In addition, some more challenging attacks at WebSockets are the Stored and Reflected Cross-Site Scripting.

At the end of the coding period I am going to commit a basic infrastructure supports active scans. I strongly believe that if we will have the basic infrastructure, we could easily add some of the above scans (as more as possible).

How To Build/Run/Debug zap and zap-extensions with InteliJ

  1. Get zaproxy & zap-extensions source code
    • First of all, keep zaproxy, zap-extension and other zap's modules under the same top directory. 
      mkdir zap-project 
cd zap-project
git clone https://github.com/zaproxy/zaproxy.git
git clone https://github.com/zaproxy/zap-extensions.git
  2. Import Project to InteliJ
    • Import zaproxy
      • File -> New -> Project From Existing Source..
      • Select .project file which is into zaproxy directory and hit next.
      • Check Create Modules files near classpath files
      • Add all the libraries that were found
      • Include all the modules that were found
      • Use JDK 1.8 as the project SDK
      • Use the web framework if you're prompted for it
    • Import zap-extension & other modules
      • Open project structure (File -> Project Structure...).
      • Select Modules and hit Add Module.
      • Import the module you want as before.
  3. Configure Output Building Paths
    • Open Project Structure and select Modules Tab
    • Select zaproxy module and go to Paths tab.
      • Select Use Module Compile Output Path
      • Output path: zap-project/zaproxy/build
    • Select zap-extensions module and go to Paths tab.
      • Select Use Module Compile Output Path
      • Output path: zap-project/zap-extension/build
    • Repeat that for every module you got, using the appropriate compile output path.
  4. Build Project with Ant
    • Open Ant Build Tool Window
    • Add the ant build files (build.xml)
      • For zaproxy: zap-project/zaproxy/build/build.xml
      • For zap-extensions: zap-project/zap-extensions/build/build.xml
    • Run tasks from Ant Build Tool Window
  5. Run/Debug configuration for zaproxy
    • Open Run/Debug Configuration
      • Add to Main Class: org.zaproxy.zap.ZAP
      • Working Directory: zap-project/zaproxy
      • Use class path of module: zaproxy
  6. Run/Debug add-ons
    • Open Run/Debug Configuration for zaproxy
      • Add the specific add-on's ant task in the before launch panel

Attack Vectors

Eavesdropping

If the communication is not encrypted, an intermediary or Man-in-the-Middle is able to read and/or change the communication. Using the wws:\\// scheme in the URL-parameter of the WebSocket(url,protocol) constructor provides protection against eavesdropping and modification.9

[INFO] How WebSocket Extension Works?

How WebSocket extension add a new Channel

When a new connection is detected in ExtensionWebSocket.onHandshakeResponse(...) the method ExtensionWebSocket.addWebSocketChannel(...) is called. It expects the HTTPMessage instance of the handshake, the socket for communication with the browser, the socket for communication with the server and the current InputStream from the server. The latest argument is of importance, as first WebSocket messages are allowed to appear in the same TCP packet after the HTTP response9.

How WebSocket Proxy works

The ExtensionWebSocket class creates a new instance of WebSocketProxy via the factor method WebSocketProxy.create(...) that returns a version specific WebSocketProxy instance. Each WebSocketProxy instance creates two instances of WebSocketListener. The instances are threads listening to one of the given sockets, either for incoming or outgoing messages. If the first byte arrives, it calls WebSocketProxy.processRead(..) that handles the received WebSocket frame. Until processing is finished, no further WebSocket frames within the responsibility of the WebSocketListener is handled. The WebSocketProxy class implements to Observer-pattern, allowing instances of WebSocketLObserver to get notified about new frame or a change of the state of a WebSocket connection. Whenever a message arrives or a part of it the WebSocketLObserver.onMessageFrame method is called. When further processing and notifications should be stopped, false can be return. When the internal state of the WebSocketProxy changes, the WebSocketLObserver.onStateChange(...) method is called.

  • Defined Stages of WebSocket API
    • CONNECTING(0): The connection has not yet been established.
    • OPEN(1): The WebSocket Connection is established and communication is possible.
    • CLOSING(2): The connection is going through the closing handshake.
    • CLOSED(3): The connection has been closed of could be not be opened
How User Interface Work?
  • MessageView

    A JTable is used to display communication details. The class making use of WebSocketMessageView. It uses a special model instance from type WebSockeMessagesViewModel. Both are initialized in the WebSocketPanel and held in the private attribute messageView & messageModel.

    The class of the messagesModel attribute is special, as it inherits from the abstract class PagingTableModel. This abstract model class is able to handle thousands of entries as it holds only PagingTableModel.MAX_PAGE_SIZE entries in memory at any given time, but the scrollbar of JTable appears as it would contain all entries.

    In case of WebSockeMessagesViewModel the database is used to query the total row count & new pages consisting of several entries. It is aware of the filter applied to the messages view and adjust the query to the database accordingly.
